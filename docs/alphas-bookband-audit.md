# 알파스 책밴드 vs 우리 interlibrary 어댑터 — 정합 audit

> **출처**: `알파스 공통 V 1.0.pdf` §15 책밴드서비스 (p.65~76, 본문 §14로 잘못 표기 — 본문 §15)
> **분석일**: 2026-04-28
> **목적**: 알파스 자체 책밴드 시스템 vs 우리 책나래·책바다·RISS export 어댑터 본질 차이 명확화

---

## 0. 핵심 차이 — 우리는 "신청서 자동 생성기", 알파스는 "도서관 간 매칭 SaaS"

| 측면 | 알파스 책밴드 | 우리 interlibrary/exporters.py |
|---|---|---|
| **위치** | 알파스 통합 SaaS 내장 (자관 시스템 일부) | KOLAS·DLS·알파스 사용 도서관의 **신청서 csv·xlsx 자동 생성** |
| **데이터 저장** | 회원 PII·자료 PII·진행상태·SMS 이력 모두 DB 저장 | **저장 0** (사서 입력 dict → csv·xlsx export만) |
| **PII 노출** | 회원·가족·전화번호·자료상태 광범위 | 사서가 입력한 dict의 컬럼 매핑만 (책나래 `이용자ID` 컬럼 등) |
| **PIPA 패턴 1** | Reader entity ERD 존재 | ✅ Reader entity ERD 부재 (생존 조건 충족) |
| **상호대차 시스템** | 책밴드 (자체) — 알파스 사용 도서관 간 | 책나래·책바다·RISS (정부) — 전국 도서관 |
| **RFID 통합** | 발송·입수·복귀·복귀완료 RFID 직접 처리 | RFID 스코프 X — csv·xlsx 양식 생성만 |

**우리 SaaS 위치**: 알파스/KOLAS/DLS를 이미 쓰는 도서관이 **책나래·책바다·RISS 신청서 작성 시간을 줄이는 도구** — 자관 시스템 대체 X.

## 1. 알파스 책밴드 8단계 워크플로우 (p.69 표)

```
신청 → 요청 → 발송 → 입수 → 책밴드대출 → 책밴드반납 → 복귀 → 복귀완료
```

### 진행상태 코드 9개 (자료별 lifecycle)
| 코드 | 상태 | 의미 |
|---|---|---|
| 0001 | 신청자료 | 이용자 또는 사서가 책밴드 신청한 직후 |
| 0002 | 요청자료 | 요청도서관이 제공도서관에 요청 송신 |
| 0004 | 발송자료 | 제공도서관이 자료 발송 처리 |
| 0014 | 입수자료 | 요청도서관이 자료 수령 |
| 0008 | 복귀자료 | 책밴드반납 후 제공도서관 복귀 대기 |
| 0009 | 완료자료 | 복귀 완료 — lifecycle 종료 |
| 0011 | 발송거절 | 제공도서관이 발송 거절 |
| 0012 | 요청거절 | 요청도서관이 자료 상태 등 사유로 요청 거절 |

### 자동화 옵션 (책밴드옵션설정)
- **신청시자동요청기능**: 신청 즉시 자동 요청 송신 (사서 개입 X)
- **SMS 자동·수동**: 발송거절·입수 시 이용자에게 SMS

## 2. 우리 interlibrary/exporters.py 4 함수 정합

### 현재 구조 (kormarc-auto v0.4.35)
```
_book_to_row()        — 단일 책 dict → CSV 행 변환
_chaeknarae_map()     — 책나래 컬럼 매핑 (국립장애인도서관 우편 서비스)
_chaekbada_map()      — 책바다 컬럼 매핑 (국가상호대차)
_riss_map()           — RISS 컬럼 매핑 (대학 상호대차)
_convert()            — 통합 변환
write_csv()           — 일반 CSV 출력
write_xlsx()          — Excel 출력 (지자체 양식)
from_inventory()      — kormarc 인벤토리 → 신청서 자동
```

### 알파스 책밴드와의 정합 매트릭스
| 알파스 기능 | 우리 정합 | 근거 |
|---|---|---|
| 신청서 데이터 입력 | ✅ `from_inventory()` 자동 | 사서 시간 절감 |
| 진행상태 추적 (0001→0009) | ❌ 미흡수 | 알파스 자체 시스템이 추적 — 우리 영역 X |
| 양방향 도서관 매칭 | ❌ 미흡수 | 정부 시스템 (책나래·책바다)이 처리 |
| RFID 통합 | ❌ 미흡수 | 자관 RFID 시스템이 처리 — 우리 영역 X |
| SMS 알림 | ❌ 미흡수 | 자관 시스템이 처리 |

**결론**: 우리는 **신청서 작성 시간 (사서 평균 5~10분/건 → 30초)** 만 단축. 책밴드 자체 시스템 구축 X.

## 3. 컴플라이언스 함의 (PIPA 패턴 1 audit)

### 알파스 책밴드 PII 처리
- 회원 (이용자) 본인 정보
- 가족 회원 정보 (가족 아이디 검색)
- 전화번호·이름 (비내방 이용자)
- 회원상태 (대출정지·책밴드신청제한)
- 자료상태 (파손·분실·대출중)
- 처리목록 이력 (lifecycle 전체 timestamp)

→ 알파스 통합 SaaS는 **PIPA 5대 패턴 모두 충족 의무** (5만명+ 도서관 회원 처리)

### 우리 PII 처리 (interlibrary/exporters.py)
- `_chaeknarae_map()` line 100: `"이용자ID": ["patron_id", "user_id"]` — **컬럼 매핑만**
- `_chaekbada_map()` line 118: `"신청자ID": ["patron_id"]` — **컬럼 매핑만**
- 입력: 사서가 작성한 dict
- 출력: csv·xlsx 파일
- **저장: 0** (DB 테이블·로그 모두 X)

→ ✅ **Reader entity ERD 부재 정합 유지** (생존 조건)

### 향후 위험 (피해야 할 것)
- ❌ 신청 이력 DB 저장 (분쟁 추적 등 명목)
- ❌ 진행상태 알림 SMS·이메일 (이용자 연락처 보관)
- ❌ 회원 정보 캐싱 (반복 입력 절감 명목)

**의사결정 원칙**: 책밴드 자체 시스템 기능을 **흉내내는 순간 PIPA 패턴 1 위반** → 매출 10% 과징금 위험 (2026-09-11 시행).

## 4. 향후 호환성 (선택)

### 가능성 1 — 알파스 책밴드 신청 양식 export
알파스 사용 도서관이 우리 SaaS로 책밴드 신청서를 생성 → 알파스 화면에 붙여넣기.
- 가치: 낮음 (알파스 책밴드는 자체 신청 화면 = 우리 우회 가치 0)
- 결정: 미진행

### 가능성 2 — 알파스 import (KORMARC export)
알파스에서 KORMARC export → 우리 SaaS로 import → 검증 후 다시 export.
- 가치: 높음 (4단 검증 엔진 활용)
- 결정: ADR 후보 — 알파스 export 양식 확보 후
