## 0. 통합 평가 공식

```
종합 점수 = 사업 5질문 × 0.6 + 기술 6차원 × 0.4
   + Q5 (컴플) 별도 게이트 (FAIL 시 즉시 폐기, 점수 무관)

Beta 단계 임계값:
- 종합 ≥ 75 → ACCEPT (자율 commit)
- 60~74 → 검토 (PO 결정)
- < 60 → 폐기
+ Q5 = FAIL → 즉시 폐기
```

## 1. 사업 5질문 (60% 가중치)

### Q1 — 결제 의향 (Beta 가중치 40)

| 라벨 | 점수 | 정의 |
|---|---:|---|
| HIGH | 80~100 | 권당 시간 절감 ≥5분 OR 결제 직접 트리거 |
| MID | 50~79 | 시간 절감 1~5분 OR 사용성 큰 개선 |
| LOW | 20~49 | "있으면 좋은" 수준 |
| 0 | 0~19 | 사서 워크플로 무관 |

```
Q1 = min(100, 권당_절감_분 × 10 + 결제전환_기여도 × 30)
권당_절감_분 = T_manual − T_auto (실측, N≥30)
결제전환_기여도 ∈ {0, 0.5, 1}
```

### Q2 — 비용 (Beta 25)

```
C_API_권당 = (입력토큰 × 3 + 출력토큰 × 15) / 1_000_000 × 1400
C_total_권당 = C_API + C_infra + C_support
임계값:
- ≤ ₩50: HIGH (마진 안전)
- ₩50~100: MID
- ₩100~200: LOW
- ≥ ₩200: 0 (폐업 위험)
```

헌법 §12 정합: 권당 ₩100 가격 → 비용 ≤₩10 → 마진 ≥90%.

### Q3 — 자산 (Beta 15)

```
Q3 = (재사용성 × 0.4) + (영업 활용 × 0.3) + (차별화 × 0.3)
```

### Q4 — 락인 (Beta 10)

| 메커니즘 | 점수 |
|---|---:|
| 자관 양식 등록 (정책 ③) | 80 |
| KORMARC 史료 누적 (.mrc archive) | 70 |
| 사서 routine 정합 | 90 |
| 통계 历사 | 70 |
| KOLAS·알파스·KERIS 양식 자동 mail merge | 60 |

### Q5 — 컴플라이언스 (★ 별도 게이트)

PIPA 5대 패턴 (시행 2026-09-11, 매출 10% 과징금):
1. Reader/Borrower/Patron entity ERD 부재
2. 암호화 (bcrypt·AES-256·TLS 1.2+)
3. DSAR (제35·36·37·35조의2)
4. 72h 신고 자동화
5. audit_log + 해시 체인

```
Q5 ∈ {PASS, FAIL}
FAIL → 즉시 폐기 (다른 점수 무관)
```

## 2. 기술 6차원 (40% 가중치, ADR 0010)

| 차원 | 정의 | 점수 |
|---|---|---:|
| OS 호환성 | Windows·macOS·Linux | -1~+1 |
| 데이터 거버넌스 | PII 외부 송신·로컬 보관 | -1~+1 |
| 보안 | 암호화·인증·OWASP | -1~+2 |
| 의존성 | 활발 유지보수·메이저 버전 안정 | -1~+1 |
| 롤백 | 단일 명령 vs 복잡 마이그 | -1~+2 |
| 관측 가능성 | logging·metrics·tracing | -1~+2 |

```
6dim = sum × (100/9)  # 정규화 (-9~+9 → 0~100)
ACCEPT 기준: 합계 ≥ +6 + 모든 차원 ≥ 0
```

## 3. 캐시카우 §0/§12 양수 영향 (필수)

매 자율 commit은 다음 둘 중 하나 이상에 양수 영향이어야 함:

| 영역 | 측정 |
|---|---|
| **§0 사서 마크 시간 단축** | 권당 8~15분 → 권당 2분 (87% 단축) |
| **§12 사서 본인 예산 결제 의향 ↑** | 캐시카우 직결 (200관 × 3.3만 = 월 660만원 목표) |

→ 평가축 음수 → commit 거부.
