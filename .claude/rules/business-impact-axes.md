## 0. 적용 원리

모든 commit·ADR·새 의존성·새 화면·새 hook 작성 시 5 질문 통과 필수.

각 질문 = 정성 4단 (HIGH/MID/LOW/0) + 정량 점수 (0~100).
단계별 가중치 곱셈 → 합계 ≥ 임계값이면 ACCEPT.

핵심 원리: **"이 코드가 사라져도 ARR이 흔들리지 않으면 코드를 쓸 이유가 없다."**
단 Q5 (컴플)만은 ARR이 아닌 **생존(survival)** 조건이므로 임계값 무관 별도 게이트.

## 1. Q1 — "이 기능이 사서에게 결제할 이유를 주는가?"

### 4단 라벨

| 라벨 | 점수 | 정의 |
|---|---:|---|
| HIGH | 80~100 | 권당 시간 절감 ≥5분 OR 결제 직접 트리거 |
| MID | 50~79 | 시간 절감 1~5분 OR 사용성 큰 개선 |
| LOW | 20~49 | "있으면 좋은" 수준 |
| 0 | 0~19 | 사서 워크플로 무관 |

### 정량 공식

```
Q1 = min(100, 권당_절감_분 × 10 + 결제전환_기여도 × 30)
권당_절감_분 = T_manual − T_auto (실측, N≥30)
결제전환_기여도 ∈ {0, 0.5, 1}  # 무관 / 보조 / 직접
```

### Beta 단계 의사결정 (현재)

| Q1 | 결정 |
|---:|---|
| 80~100 | 즉시 |
| 50~79 | 검토 |
| 20~49 | 백로그 |
| 0~19 | 폐기 |

## 2. Q2 — "이 코드가 비용을 늘리는가, 줄이는가?"

### 권당 비용 산정

```python
C_API_권당 = (입력토큰 × 3 + 출력토큰 × 15) / 1_000_000 × 1400  # ₩1,400/$, Sonnet 4.6
C_infra_권당 = 월_인프라비 / 월_처리권수
C_support_권당 = (지원티켓률 × 응답시간_분 × 사서시급) / 권수
C_total_권당 = C_API + C_infra + C_support
```

### 임계값

| 권당 비용 | 평가 |
|---:|---|
| ≤ ₩50 | HIGH (마진 안전) |
| ₩50~100 | MID (마진 압박) |
| ₩100~200 | LOW (위험) |
| ≥ ₩200 | 0 (폐업 위험) |

### 헌법 §12 정합

- 가격 권당 ₩100 → 비용 ₩7~10 → 마진 ≥ 90% 목표

## 3. Q3 — "이 코드가 자산이 되는가?"

### 자산 카테고리

| 자산 | 측정 |
|---|---|
| 도메인 데이터 | NLK 시소러스·전거 활용 정도 |
| 검증 엔진 | 4단 검증 정합률 (자관 .mrc 174 = 100% ✅) |
| 영업 자료 | 정부 인용·자관 PILOT·NPS |
| 차별화 코드 | NLK validator 부재 시장 진입 |
| 사서 교육 | 우리 SaaS 사용 학습 곡선 ↓ |

### 점수 공식

```
Q3 = (재사용성 × 0.4) + (영업 활용 × 0.3) + (차별화 × 0.3)
재사용성 ∈ [0~100]: 1회성 X · 다른 모듈에서 호출됨 100
영업 활용 ∈ [0~100]: 영업 자료 인용 가능 여부
차별화 ∈ [0~100]: 경쟁사 부재 영역 진입
```

## 4. Q4 — "이 코드가 락인을 만드는가?"

### 락인 메커니즘

| 메커니즘 | 점수 |
|---|---:|
| 자관 양식 등록 (정책 ③) | 80 |
| KORMARC 史료 누적 (.mrc archive) | 70 |
| 사서 routine 정합 (자관 5년 책단비·3년 정시 캡처·1년 40 차수) | 90 |
| 통계 历사 (자관 6년 NPS) | 70 |
| KOLAS·알파스·KERIS 양식 자동 mail merge | 60 |
| 직원 교육 자료 통합 | 50 |

### 임계값

| Q4 | 평가 |
|---:|---|
| 80~100 | HIGH (이전·전환 비용 ↑) |
| 50~79 | MID |
| < 50 | LOW (전환 자유) |

## 5. Q5 — "이 코드가 컴플라이언스를 위반하는가?" (★ 별도 게이트)

### PIPA 5대 패턴 (생존 조건)

| 패턴 | 위반 시 |
|---|---|
| 1. Reader/Borrower/Patron entity ERD 부재 | 매출 10% 과징금 (시행 2026-09-11) |
| 2. 암호화 (bcrypt·AES-256·TLS 1.2+) | 시행령 §7 |
| 3. DSAR (제35·36·37·35조의2) | 정보주체 권리 |
| 4. 72h 신고 자동화 | 1,000명+·민감정보·해킹 시 |
| 5. audit_log + 해시 체인 | 5만명+·민감정보 처리 |

### 카카오 학습 (151억 과징금)

> "부분 적용 후 기존 데이터 미마이그레이션이 결정타"

→ 보안 패치 시 전수 마이그레이션 강제.

### 게이트 (별도)

```
Q5 ∈ {PASS, FAIL}
- PASS: 5 패턴 모두 정합 (or 영역 X)
- FAIL: 1 이상 위반 → 즉시 폐기 (Q1·Q2·Q3·Q4 점수 무관)
```

## 6. 단계별 가중치 (Beta 단계 = 현재)

| 단계 | Q1 | Q2 | Q3 | Q4 | Q5 |
|---|---:|---:|---:|---:|---:|
| MVP | 30 | 20 | 10 | 15 | 25 |
| **Beta (현재)** | **40** | **25** | **15** | **10** | **10** |
| Payment Launch | 25 | 30 | 20 | 15 | 10 |
| Stable | 20 | 25 | 25 | 20 | 10 |

### 합계 임계값

```
합계 = Q1 × w1 + Q2 × w2 + Q3 × w3 + Q4 × w4 + Q5_게이트
Beta 단계 임계값:
- 합계 ≥ 70 → ACCEPT
- 50~69 → 검토 (자율 X·PO 결정)
- < 50 → 폐기
+ Q5 = FAIL → 즉시 폐기 (점수 무관)
```

## 7. 6 차원 평가 (기술) 통합

ADR 0010 6차원 (OS·데이터·보안·의존성·롤백·관측) × 0.4 + 사업 5질문 × 0.6 = 종합 평가.

### 종합 임계값 (Beta)

```
종합 = 6dim × 0.4 + 5q × 0.6
Beta:
- 종합 ≥ 75 → ACCEPT
- 60~74 → 검토
- < 60 → 폐기
```

## 8. ADR 0013 active 후 적용

본 rules는 ADR 0013 (사업 5질문 도입) PO 결정 후 active.

### 자율 commit 게이트

매 commit message에 명시:

```
Q1 = N (점수)
Q2 = N
Q3 = N
Q4 = N
Q5 = PASS|FAIL
종합 = N
```

### 자율 거부 사례

- Q1 < 50 + 다른 합계 부족 → 거부
- Q5 = FAIL → 즉시 거부 (다른 점수 무관)
